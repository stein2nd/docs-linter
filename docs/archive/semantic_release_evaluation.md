# 📘 S2J Docs Linter - セマンティック・リリース評価

## 設計意図 (ゴール)

下記の目的で、S2J Docs Linter におけるセマンティック・リリース採用可否を評価し、リリース自動化のを明確化します。

* リリースガバナンスの明確化
* 保守性の向上
* 自動化導入の可否判断
* 将来の拡張方針の整理

## 評価対象

* 評価対象
    * セマンティック・リリース
* 評価観点
    * リリース自動化の適性
    * 運用の複雑さ
    * 保守の負担
    * 現在のワークフローとの互換性
    * コントリビューターのワークフローへの影響

## 現在のリリース体制

S2J Docs Linter は、既に下記を備えています。

* npm Trusted Publishing (OIDC)
* GitHub Actions によるタグベースの公開
* セマンティックバージョニングのガバナンス
* バージョン管理ポリシー
* GitHub Release による自動生成ノート
* リリースドキュメント

現在のリリーストリガーは、下記の順番になります。

```zsh
git tag vX.Y.Z
git push origin vX.Y.Z
```

## セマンティック・リリースのメリット

期待されるメリットには、下記が挙げられます。

* 完全なリリース自動化
    * version bump
    * npm publish
    * GitHub リリース
    * changelog 生成
* 手動によるリリース手順の削減
    * 手動操作の削減。
* 標準的なリリース手順
    * コミット規約に基づいたリリースの判定。

## セマンティック・リリースのコスト

* 構成ワークフローに、下記が追加されることで、複雑さが増す。
    * セマンティック・リリースの設定
    * プラグインのメンテナンス
    * CI ワークフローの変更
* コミット規約の結合により、既存のコントリビューターのワークフローに影響する。
    * 通常、下記の様な従来のコミットが前提。
        * feat:
        * fix:
        * breaking:
* 下記に依存する為、メンテナンスのオーバーヘッドが増す。
    * セマンティック・リリースのエコシステム
    * プラグインの互換性
    * 設定のメンテナンス
* 人的制御の著しい低下。
    * 現在…意図的なタグによるリリース
    * セマンティック・リリース…自動化によるリリース

## S2J Docs Linter との適合性

評価は、下記になります。

* 現在のプロジェクト規模
    * S2J Docs Linter は、小〜中規模 OSS ツールである。
    * 現在のリリース頻度は、限定的であると言える。
* 既存の自動化の充足度
    * 既に下記を満たしていて、不足は限定的であると言える。
        * OIDC publish
        * GitHub リリースノート
        * SemVer ガバナンス
* コントリビューターの利便性
    * 現在の下記ワークフローは、十分にシンプルであると言える。
        ```zsh
        npm version patch
        git push
        git push --tags
        ```

## 判断

下記の理由から、「現時点では非採用」との結論になります。

* 現在のワークフローで十分である
* セマンティック・リリースは不必要な複雑さを招く
* 貢献者の負担が増大する
* メンテナンスコストがメリットを上回る

## 再評価条件

下記の様な状況になった時点には、再評価します。

* リリース頻度が大幅に増加する
* 貢献者の数が増える
* 手動によるリリースの負担が顕在化する
* changelog のメンテナンス負担が増加する
* リリースガバナンスの複雑さが増す

## 方針

* 現在の標準…手動での SemVer + GitHub Actionsによる OIDC publish
* セマンティック・リリース…将来的な検討の余地あり

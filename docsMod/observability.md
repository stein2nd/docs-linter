# 📘 S2J Docs Linter - SLA (サービスレベル契約)・指標・トレース

## 1. 可観測性の仕様

本書は、S2J Docs Linter プラットフォームの可観測性の契約を定義します。

可観測性は、プラットフォームのランタイム状態を可視化し、性能/健全性/信頼性を継続的に評価するための契約です。

対象は、下記のコンポーネントとします。

* Core ランタイム
* REST ランタイム
* SDK
* SDK ジェネレーター
* 「利用側」連携

## 2. 目的

可観測性は、下記を目的とします。

* ランタイム・ヘルスの把握
* パフォーマンス・バジェットの維持
* 失敗解析
* 継続的改善
* 運用可視性

## 3. 可観測性の原則

プラットフォームは、下記の原則に従います。

* デフォルトで観測可能
* 指標 First
* 構造化ログ記録
* 追跡可能な実行
* オーバーヘッドの少ないモニタリング

## 4. パフォーマンス・バジェット

ランタイムは、「パフォーマンス・バジェット」を満たすことを目標とします。

### バジェット・カテゴリー

* 起動時間
* リクエスト・レイテンシー
* 検証スループット
* メモリー使用量
* CPU 使用率

### ルール

パフォーマンス・バジェットは、ランタイム・プロファイルごとに定義できます。

## 5. 契約

### ログ記録契約

ランタイムは、構造化ログを生成します。

### トレース契約

リクエストは、トレース可能であることとします。

### Advanced 可観測性の契約

可観測性の契約は、ランタイム、デプロイメント、および互換性と連携し、プラットフォーム全体の「運用の可視性」を保証します。

### テレメトリ契約

テレメトリは、ランタイムの状態を表現する標準契約です。

### 相関契約

テレメトリは、ランタイム間で関連付けられます。

### SLO (サービスレベル目標) / SLA (サービスレベル契約) の契約

ランタイムの品質目標を定義します。

### 診断契約

ランタイムは、診断情報を提供できます。

### イベント契約

ランタイムは、標準イベントを発行できます。

### 可観測性の互換性契約

可観測性の契約の互換性を、管理します。

## 6. レイテンシー契約

### 標準指標

* 平均レイテンシー
* P95 レイテンシー
* P99 レイテンシー

### ルール

レイテンシーは、継続的に収集します。

## 7. スループット契約

### 指標

* 秒間のリクエスト
* 秒間の検証
* 一括スループット

## 8. リソース利用契約

### 指標

* CPU
* メモリー
* ファイル・ハンドル
* スレッド

### ルール

ランタイムは、リソース使用状況を公開できます。

## 9. ログ記録

ランタイムは、構造化ログを生成します。

### ログ・レベル

* Trace
* Debug
* Info
* Warn
* Error

### ルール

Secret および個人情報を、出力してはなりません。

## 10. 指標契約

### 必須指標

* 検証回数
* エラー回数
* キャッシュヒット率
* 起動時間
* アクティブ・ランタイム

### ルール

指標は、機械可読で取得できることとします。

## 11. トレース

リクエストは、トレース可能であることとします。

### 標準属性

* トレース ID
* リクエスト ID
* ランタイム
* プロファイル
* バージョン

### ルール

トレース ID は、ランタイムを跨いで維持します。

## 12. ヘルス契約

### 必須の状態

* 起動
* 準備完了
* 正常
* 劣化
* 不健全

### ルール

ヘルスは、機械可読で公開します。

## 13. アラート契約

### アラートの対象

* エラー率
* タイムアウト
* 起動失敗
* リソース枯渇

### ルール

重大アラートは、通知対象とします。

## 14. パフォーマンス回帰の方針

パフォーマンス回帰を検出します。

### 必須テスト

* ベンチマーク
* 回帰テスト
* 負荷テスト

### ルール

パフォーマンス・バジェットを超えた変更は、レビュー対象とします。

## 15. ランタイム・プロファイル指標

### 標準プロファイル

* ブラウザー
* WordPress
* CLI
* REST

### ルール

プロファイルごとに、異なるバジェットを持てます。

## 16. Advanced 可観測性

本章は、可観測性仕様を補完します。

本章では、ランタイムの可観測性を機械可読かつ継続的に評価するための契約を定義します。

可観測性の契約は、ランタイム、デプロイメント、および互換性と連携し、プラットフォーム全体の「運用の可視性」を保証します。

## 17. 可観測性マニフェスト

各ランタイムは、「可観測性マニフェスト」を公開します。

可観測性マニフェストは、ランタイムが提供するモニタリング機能を記述します。

下記は、可観測性マニフェスト例です。

```yaml id="y4rm7t"
observabilityVersion: "1.0"

metrics:
  enabled: true

logging:
  level: info

tracing:
  enabled: true

sampling:
  strategy: adaptive
```

### 必須プロパティ

* マニフェスト・バージョン
* 指標設定
* ログ記録設定
* トレース設定
* サンプリング方針
* ランタイム・プロファイル

### ルール

可観測性マニフェストは、デプロイメント・マニフェストから参照可能であることとします。

## 18. テレメトリ

テレメトリは、ランタイムの状態を表現する標準契約です。

### テレメトリの種類

* 指標
* ログ
* トレース
* イベント

### ルール

テレメトリ・フォーマットは、ランタイムに依存してはなりません。

## 19. 相関

テレメトリは、ランタイム間で関連付けられます。

### 標準識別子

* トレース ID
* リクエスト ID
* 相関 ID
* ランタイム ID

### ルール

相関 ID は、ランタイムを跨いで維持します。

## 20. 指標プロファイル

ランタイムごとに「指標プロファイル」を定義します。

### 標準プロファイル

* ブラウザー
* WordPress
* CLI
* REST
* SDK ジェネレーター

### 標準指標

* 起動時間
* リクエスト数
* エラー数
* メモリー使用量
* CPU 使用率
* 検証スループット

### ルール

プロファイルごとに「パフォーマンス・バジェット」を設定できます。

## 21. SLO (サービスレベル目標) / SLA (サービスレベル契約)

ランタイムの品質目標を定義します。

### 標準目標

* 可用性
* レイテンシー
* エラー率
* 起動時間

### ルール

パフォーマンス・バジェットは、SLO (サービスレベル目標) と整合することとします。

## 22. 診断

ランタイムは、診断情報を提供できます。

### 標準情報

* ランタイム・バージョン
* プロファイル
* 機能
* 設定の概要
* ヘルス・ステータス

### ルール

Secret および個人情報を、診断に含めてはなりません。

## 23. イベント

ランタイムは、標準イベントを発行できます。

### 標準イベント

* RuntimeStarted
* RuntimeStopped
* ValidationCompleted
* ValidationFailed
* DeploymentCompleted
* HealthChanged

### ルール

イベントは、機械可読であることとします。

## 24. 可観測性プロファイル

環境ごとのモニタリング方針を定義します。

### プロファイル

#### 開発

* 詳細なログ記録
* フル・トレース
* デバッグ指標

#### テスト

* ベンチマーク指標
* 契約指標

#### 本番

* 構造化ログ記録
* 最適化された指標
* 適応型トレース

### ルール

本番プロファイルは、低オーバーヘッドを優先します。

## 25. サンプリング方針

トレースおよびイベントの収集方法を定義します。

### サポート対象の戦略

* 常時 On
* 固定レート
* 適応型
* エラーのみ

### ルール

サンプリング戦略は、ランタイム・プロファイルにより変更できます。

## 26. 可観測性の互換性

可観測性の契約の互換性を、管理します。

### 互換性の対象

* マニフェスト・バージョン
* 指標契約
* イベント契約
* テレメトリ契約

### ルール

可観測性の契約の破壊的変更は、メジャー・バージョンのみ許可します。

## 27. 横断原則

### デフォルトで可観測

ランタイムは、標準で可観測性を提供します。

### 低オーバーヘッド

モニタリングは、ランタイムの性能を著しく低下させてはなりません。

### 機械可読

可観測性情報は、機械可読で提供します。

### 継続的な測定

パフォーマンスは、継続的に評価します。

## 28. 完了条件

可観測性の仕様は、下記を実装した時点で完成とみなします。

* 可観測性の原則
* パフォーマンス・バジェット
* レイテンシー契約
* スループット契約
* リソース利用契約
* ログ記録契約
* 指標契約
* トレース契約
* ヘルス契約
* アラート契約
* パフォーマンス回帰の方針
* ランタイム・プロファイル指標
* ADR (アーキテクチャ決定記録)

* 可観測性マニフェスト
* テレメトリ契約
* 相関契約
* 指標プロファイル
* SLO (サービスレベル目標) / SLA (サービスレベル契約) の契約
* 診断契約
* イベント契約
* 可観測性プロファイル
* サンプリング方針
* 可観測性の互換性契約
* 横断原則
* Advanced 可観測性の契約 ADR (アーキテクチャ決定記録)

## 29. ADR (アーキテクチャ決定記録)

### ADR-OBS-001

#### タイトル

* 指標 First

#### 決定

* 指標を、可観測性の中心とする。

### ADR-OBS-002

#### タイトル

* 構造化ログ記録

#### 決定

* ログは、構造化フォーマットとする。

### ADR-OBS-003

#### タイトル

* パフォーマンス・バジェット

#### 決定

* パフォーマンス・バジェットを、契約化する。

### ADR-OBS-004

#### タイトル

* 分散トレース

#### 決定

* トレース ID を、ランタイム間で共有する。

### ADR-OBS-005

#### タイトル

* ヘルス駆動オペレーション

#### 決定

* ヘルス・ステータスを、ランタイムの標準契約とする。

## 30. Advanced 可観測性の契約 ADR (アーキテクチャ決定記録)

### ADR-OBS-006

#### タイトル

* 可観測性マニフェスト

#### 決定

* ランタイムは、「可観測性マニフェスト」を公開する。

### ADR-OBS-007

#### タイトル

* 統一テレメトリ

#### 決定

* テレメトリを、共通契約として定義する。

### ADR-OBS-008

#### タイトル

* 相関 First

#### 決定

* トレース、リクエスト、およびランタイムを、相関 ID により関連付ける。

### ADR-OBS-009

#### タイトル

* プロファイル・ベースのモニタリング

#### 決定

* モニタリングは、ランタイム・プロファイルにより切り替える。

### ADR-OBS-010

#### タイトル

* パフォーマンス駆動のオペレーション

#### 決定

* パフォーマンス・バジェットと SLO (サービスレベル目標) を、一体で管理する。

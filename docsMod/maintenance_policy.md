## S2J Docs Linter - メンテナンス方針

### 設計意図 (ゴール)

S2J Docs Linter (`@s2j/docs-linter`) を、継続的に保守可能な npm パッケージとして運用するためのメンテナンス方針を定義します。

本方針の目的は、下記の通りです。

* パッケージ品質の維持
* リリースプロセスの安定化
* 依存関係の健全性の維持
* セキュリティ態勢の継続的な改善
* 貢献者とメンテナー間の判断基準の統一

### 設計原則

* 利用側プロジェクトの継続運用を最優先とすること。既存利用者に、不要な破壊的変更を持ち込まない。
* リリース / ビルド / パッケージングは、再現可能であること。ローカル・CI・npm レジストリ間で一貫性を保つ。
* npm publish / CI/CD / 認証については、「デフォルトで安全であること」を原則とする。長期 secret の利用を最小化する。
* 運用判断を暗黙知にしない。リリース / 互換 / 非推奨の判断基準を、文書化する。
* 個人プロジェクトであっても、OSS パッケージとして最低限の保守性を担保する。

### 設計方針 (規約)

* リリース管理
    * リリースは、npm パッケージとして publish 可能な状態を維持する。
    * リリースの標準経路は下記とし、手動 publish は、代替手段とする。
        * GitHub Actions
        * npm Trusted Publishing (OIDC)
* バージョン管理
    * バージョン管理には、セマンティック・バージョニングを採用する。
    * 原則は、下記の通りとする。
    * PATCH
        * バグ修正
        * ドキュメント lint ルールの修正
        * 互換性を損なわない内部リファクタリング
    * MINOR
        * プリセットの拡張
        * 新しい CLI 機能
        * 後方互換性を維持した機能強化
    * MAJOR
        * プリセットレイアウトの互換性を破る変更
        * CLI の互換性を破る変更
        * Node.js の互換性を破る変更
* 互換性
    * 最低サポート runtime は、`Node.js >= 20` とする。
    * package.json の `engines.node` を source of truth とする。
* CI メンテナンス
    * 下記の CI を継続運用対象とする。CI 障害は、保守上の問題とみなす。
        * パッケージの検証
        * npm publish ワークフロー
        * docs lint ワークフロー
        * Node.js 互換性の検証
* 依存関係の管理
    * npm の下記に関する依存関係は、継続的に確認する。古い依存関係の放置を避ける。
        * 製品の依存関係
        * 開発の依存関係
        * GitHub Actions の依存関係
* セキュリティの維持
    * publish 認証は、Trusted Publishing を標準とし、下記は禁止する。
        * GitHub Secrets への恒久 `NPM_TOKEN`
        * 平文での認証情報のコミット
        * セキュリティ対策が不十分な publish 自動化
* ドキュメントの保守
    * 下記文書を、継続的な保守対象とする。実装変更時は、関連文書の更新を推奨する。
        * README.md
        * docsMod/status.md
        * docsMod/release.md
        * docsMod/npm_usage.md
        * docsMod/npm_auth_secret_manage_spec.md
        * docsMod/maintenance_policy.md
* マイグレーション支援
    * npm パッケージの移行ガイドラインは維持する。
    * 従来の Git サブモジュール運用は、後方互換性として扱い、新規導入方法としては推奨しない。

### 非対象 (Out of Scope)

* エンタープライズ SLA
* 有償サポート
* 応答時間の保証
* 商用保守契約
* サードパーティ製依存コンポーネントのベンダー保証

### 責務

* パッケージのライフサイクルを管理すること
* リリースを管理すること
* 依存関係を整備すること
* CI をメンテナンスすること
* 互換性ポリシー
* セキュリティをメンテナンスすること
* ドキュメントをメンテナンスすること

### 非責務

* 課題への即時対応
* 将来にわたる、厳格な下位互換性の維持
* 旧バージョンの Node.js に対する、永続的なサポート
* 外部の textlint エコシステムにおける、あらゆる変更への対応

### メンテナンス対象アーティファクト

メンテナンス対象は、`@s2j/docs-linter` とし、下記を含みます。

* CLI
* presets/base
* presets/swift
* presets/wordpress
* 互換レイアウト
* GitHub Actions ワークフロー
* npm パッケージのメタデータ

### メンテナンスレビューのトリガー

下記をレビューのトリガーとします。

* npm publish の失敗
* CI の失敗
* 依存関係の脆弱性の検出
* Node.js LTS  のライフサイクル変更
* textlint のメジャーバージョン変更
* GitHub Actions の非推奨化
* ユーザー側の互換性課題

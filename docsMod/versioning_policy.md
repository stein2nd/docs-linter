## S2J Docs Linter - SemVer 方針

下記について定義します。

* MAJOR
* MINOR
* PATCH
* 破壊的変更
* 非推奨
* リリースタグのルール



### 設計意図 (ゴール)

下記を目的に、S2J Docs Linter (`@s2j/docs-linter`) のバージョニングに関する判断基準を明文化し、リリース判断の一貫性を確保します。

* セマンティックバージョニングの一貫した運用
* リリース判断の属人化防止
* コンシューマー互換性の、予測可能性の向上
* 破壊的変更の明確化
* 保守性の向上

### 設計原則

* 利用側がバージョン番号から、変更影響を予測できること。
* 不要な破壊的変更を避けること。
* 破壊的変更は、意図的・明示的に扱うこと。暗黙の破壊的変更を避ける。
* バージョン決定の基準を「パッケージ作成者」ではなく「ユーザーへの影響」とすること。
* セマンティックバージョニング (SemVer) を標準とすること。

### 設計方針 (規約)

* バージョン管理基準
    * 標準は、`Semantic Versioning 2.x`
    * 形式は、下記の様に `MAJOR.MINOR.PATCH` とする
        * 1.0.12
        * 1.1.0
        * 2.0.0
* PATCH リリース
    * PATCH は、1.0.12→1.0.13のような、下記を対象とした、下位互換性を保つ修正とする。
        * バグ修正
        * typo 修正
        * ドキュメントの修正
        * 内部リファクタリング
        * CI ワークフローの修正
        * パッケージングの修正
        * メタデータの修正
        * 互換性ミラーの修正
* MINOR リリース
    * MINOR は、1.0.12→1.1.0のような、下記を対象とした、下位互換性のある機能強化とする。
        * 新しいプリセット
        * 新しい lint ルール
        * オプションの CLI 機能強化
        * 後方互換性のある機能の追加
        * ドキュメント機能の拡張
        * 新しい連携機能のサポート
* MAJOR リリース
    * MAJOR は、1.0.12→2.0.0のような、下記を対象とした、破壊的変更を含むリリースとする。
        * プリセットレイアウトの破壊的変更
        * CLI の動作変更
        * 設定パスの非互換性
        * Node.js の最小バージョン引き上げ
        * 非推奨機能の削除
        * textlint 統合の、互換性のない変更

### 非対象 (Out of Scope)

* セマンティック・リリースの自動化
* changelog の生成方針
* GitHub リリースノートの書式設定
* エンタープライズ互換性 SLA

### 責務

* バージョン番号の増分決定
* 破壊的変更の分類
* リリースタグ付けのルール
* 互換性の解釈

### 非責務

* 永続的な下位互換性
* 将来にわたって互換性を損なわないエコシステム
* 依存関係ベンダーの動作制御

### 破壊的変更の定義

下記を破壊的変更と見做します。

* CLI に関する変更
    * 例
        * コマンド名の変更
        * 必須引数の変更
        * 出力仕様の変更
    * 例えば `s2j-docs-linter --config` の様な仕様破壊
* プリセットレイアウトに関する変更
    * 変更前が `presets/base/` で、変更後が `configs/base/` の様な、ユーザー設定が壊れる変更。
* 設定の互換性に関する変更
    * 変更前が `./node_modules/@s2j/docs-linter/presets/base/.textlintrc.base.json` で、変更後が互換なし
* ランタイム互換性に関する変更
    * 変更前が `Node >=20` で、変更後が `Node >=22`
* 依存関係の互換性に関する変更
    * 下記の様な、主要な依存関係のアップグレードにより、ユーザーに影響が生じる場合。
        * textlint エコシステムにおける破壊的変更

### 破壊的変更ではないもの

以下は、原則 PATCH / MINOR とします。

* README の更新
* 内部ビルドのリファクタリング
* GitHub Actions ワークフローの改善
* publish 認証のマイグレーション
* ドキュメントの文言変更
* 互換性を維持した依存関係の更新

### 非推奨ポリシー

機能の廃止の前に、非推奨周知を推奨します。ただし小規模な OSS では、保守性を優先し、省略可能とします。

推奨プロセスは、下記の通りです。

1. 非推奨の告知
2. 互換性維持の期間
3. MAJOR リリースでの機能削除

### リリースタグ方針

Git タグは、バージョンと一致させます。

形式は `vX.Y.Z` とし、下記の様に利用します。

```bash
git tag v1.0.13
git push origin v1.0.13
```

`release-1`、`latest-fix`、`hotfix` の様な形式は禁止とします。

### 先行公開の方針

形式は `1.1.0-beta.1`、`1.1.0-rc.1` とします。
将来必要時のみとし、フェーズ4時点では標準運用の対象外とします。

### ロールバック方針

npm publish は、不変性を前提としています。

* 原則
    * `1.0.13 overwrite` の様に、再 publish しない
    * `1.0.14 fix release` の様に、バージョンを増やして修正リリース

### 消費者の互換性に関する責任

メンテナーは、下記を考慮します。

* 設定パスの安定性
* CLI 契約の安定性
* Node.js 互換性の明確化
* マイグレーションガイド
